06Full text of the ruling decision tree

Reuse this verbatim for the next harvest. Order = cost: ask what's mechanically answerable first, save judgment calls for last.

5-way decision tree · copy verbatim

Judge each exhaust BASE NAME (strip the -wf_xxxxxxxx suffix; N files of one base = N runs) in this order, cheapest test first. Tags are labels, NEVER deletions.
Q1 internalized — does its meta.name/description match a saved recipe in the repo's canonical set (same pipeline, the recipe IS its generalization)? → tag internalized, note which recipe covers it.
Q2 preset — is it ≤20 lines, spawns zero agents itself, and just calls workflow() with baked args? → tag preset (a remembered invocation, candidate for a PRESET comment block on the recipe it wraps).
Q3 generalization candidate — do ≥2 base names share an isomorphic pipeline shape not covered by any repo canonical recipe? First re-check coverage: if an existing recipe covers the shape, reclassify internalized. Otherwise → tag generalization candidate with the shared shape named.
Q4 doctrine — is the reusable part not the pipeline but repeated discipline TEXT in the agent prompts (anchoring, tool mapping, scope fences, report contracts)? → tag doctrine, name the _lib/worker-doctrine.md block it feeds.
Q5 genuinely one-off — everything else: project-specific, single-run, not worth abstracting. → tag one-off.
